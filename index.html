How I actually engineer

The questions every hiring manager asks. Answered with specifics, not buzzwords.

How do you handle complex state?

I separate server state from UI state by default. TanStack Query manages all server data — caching, background refetching, optimistic updates. Redux (or Zustand) only enters when there's genuinely shared client state that multiple unrelated components need. I avoid putting server data in Redux; it creates two sources of truth that drift.

TanStack QueryReduxSeparation of concerns
How do you structure a large React codebase?

Feature-based folders, not type-based. features/invoices/ contains its components, hooks, types, and API calls together. Shared primitives live in components/ui/. This means when a feature changes, you touch one folder — not files scattered across components/, hooks/, utils/, types/.

Feature foldersCo-locationScalable architecture
How do you handle RBAC in a frontend app?

Permissions come from the auth token on login and get stored in context. I build a usePermissions() hook that components consume. Route guards check role at the router level. Individual UI elements (buttons, table actions) conditionally render based on permission flags. The backend still enforces everything — the frontend RBAC is purely UX.

Auth contextRoute guardsConditional rendering
How do you optimize a slow React app?

First I profile, not guess. React DevTools Profiler shows which components are re-rendering and why. Typical fixes: memoize expensive computations with useMemo, stabilize callbacks with useCallback, avoid prop drilling that forces wide re-renders, virtualize long lists with react-window. In one case at TechNationals, moving a filter calculation out of render and into a memoized selector cut re-renders from 40/second to 4.

React ProfileruseMemo / useCallbackList virtualization
How do you handle offline data in the IMS?

The Inventory System runs sql.js (SQLite compiled to WASM) entirely in-browser/Electron. On app load, it restores the database from a persisted binary file. All writes go to the in-memory SQLite instance immediately, then flush to disk. No network calls, no sync conflicts. It's a single source of truth that happens to live on the user's machine.

SQLite / sql.jsElectronOffline-first architecture
What's a hard bug you solved?

In a dashboard at TechNationals, a filter UI was causing the entire page to remount on every change instead of just re-rendering. Traced it to a context provider that was re-creating its value object on every parent render, which React treated as a new reference, invalidating every consumer. Fixed by memoizing the context value. Subtle, no console errors — just sluggish UI until you profile it.
